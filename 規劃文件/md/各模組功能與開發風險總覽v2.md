# **Bombus 企業管理系統 V6.0 \- 模組功能與開發風險總覽 (詳細版)**

**文件版本**：V6.0 (Detail Level)

**文件日期**：2025-11-21

**用途**：專案管理、Sprint Planning、風險評估

**撰寫人**：Gemini (資深系統設計規劃師)

## **1\. L0 系統核心與儀表層 (System Core)**

| 編號 | 細項功能 | 開發重點 (Key Implementation) | 潛在風險與痛點 (Risks & Pain Points) |
| :---- | :---- | :---- | :---- |
| **L0.0** | **企業管理儀表板** | **資料聚合 (Data Aggregation)**：實作快取層 (Redis) 以處理 L1-L6 跨模組數據撈取。 **動態渲染**：前端需根據 User Role 動態載入 Widget 元件。 | **效能瓶頸**：若無快取，首頁載入時間可能隨資料量暴增。 **資訊過載**：若未適當分流，高階主管可能會被瑣碎資訊淹沒。 |
| **L0.1** | **系統管理與權限控制** | **列級權限 (Row-Level Security)**：在 Middleware 層注入 WHERE dept\_id \= ? 自動過濾。 **自動備份**：整合 NAS SDK 實作每日排程與錯誤重試機制。 | **權限漏洞**：複雜的 RBAC \+ 列級權限容易產生邏輯漏洞，需單元測試覆蓋。 **備份中斷**：NAS 連線不穩時若無 Alert，可能導致資料遺失風險。 |

## **2\. L1 員工管理模組 (Employee Management)**

| 編號 | 細項功能 | 開發重點 (Key Implementation) | 潛在風險與痛點 (Risks & Pain Points) |
| :---- | :---- | :---- | :---- |
| **L1.1** | **招募與候選人管理** | **AI Pipeline**：串接 STT (語音轉字) \-\> NLP (關鍵字提取) \-\> Scoring (權重計算)。 **104 整合**：實作 RPA 或 API 定期同步履歷。 | **AI 準確度**：面試錄音品質差或口音重會嚴重影響評分。 **隱私法規**：錄音/錄影需確保符合個資法 (GDPR/PDPA) 規範。 |
| **L1.2** | **員工檔案與歷程管理** | **ROI 運算**：建立 Cost (薪資+福利) vs Value (專案+績效) 的計算公式。 **版本控制**：職務異動需保留完整的 History Log。 | **數據敏感性**：薪資數據外洩是最高風險，需資料庫層級加密。 **ROI 定義爭議**：行政職等非營收單位的「產出價值」難以量化。 |
| **L1.3** | **人才庫與再接觸管理** | **向量搜尋 (Vector Search)**：將候選人技能轉為 Vector，以便與新職缺進行模糊匹配。 **排程通知**：定期掃描冷名單並發送關懷信。 | **資料陳舊**：人才庫資料久未更新，聯繫方式可能失效。 **打擾感**：過度自動化的再行銷可能被視為騷擾郵件。 |
| **L1.4** | **職涯晉升與接班規劃** | **狀態機 (State Machine)**：精確控制「申請→試行→正式→退場」的狀態流轉。 **規則引擎**：每月 1 號自動執行績效檢核 (任務率/協作分)。 | **公平性爭議**：自動化退場機制若無人工覆核，可能引發勞資糾紛。 **邊界情況**：請長假期間的試行期計算邏輯需明確定義。 |
| **L1.5** | **會議管理** | **雙向同步**：實作 Google Calendar Webhook，處理會議變更衝突。 **Action Item 追蹤**：與 L4 任務系統連動。 | **API 配額**：頻繁同步可能觸發 Google API Rate Limit。 **同步延遲**：外部行事曆更新可能會有數分鐘延遲。 |

## **3\. L2 職能管理模組 (Competency Management)**

| 編號 | 細項功能 | 開發重點 (Key Implementation) | 潛在風險與痛點 (Risks & Pain Points) |
| :---- | :---- | :---- | :---- |
| **L2.1** | **職等職級管理** | **樹狀結構 (Tree Structure)**：資料庫設計需支援無限層級或彈性調整。 **文件生成**：整合 Office Open XML SDK 自動產出 Word 辦法。 | **矩陣複雜度**：大型組織的職等對應可能過於複雜，導致 UI 難以呈現。 **文件版本混亂**：自動生成的文件若無版控，易造成管理困擾。 |
| **L2.2** | **職務說明書管理 (JD)** | **Prompt Engineering**：調校 LLM 以生成符合 ISO 標準的 JD 格式。 **模板管理**：提供彈性的欄位自定義功能。 | **AI 幻覺**：AI 生成的 JD 可能包含不切實際的要求，需人工審核。 **格式跑版**：匯出不同格式 (PDF/Word) 時的排版一致性。 |
| **L2.3** | **職能框架開發** | **KSA 模型**：建立知識、技能、態度的正規化資料表。 **連動觸發**：評估完成後自動觸發 L3 課程推薦 API。 | **過度設計**：職能細項定義過細，會導致系統維護成本過高。 **資料孤島**：若未與 L3/L5 打通，職能框架將淪為紙上談兵。 |
| **L2.4** | **職能評估系統** | **Workflow 引擎**：處理「自評-\>主管評-\>核決」的簽核流程。 **進度催收**：自動發送評估逾期提醒。 | **評估疲勞**：問卷過長會導致員工隨意填寫，數據失真。 **主觀偏差**：缺乏客觀數據佐證，評分容易流於印象分數。 |
| **L2.5** | **職能落差分析** | **Gap 演算法**：計算 Standard \- Actual 並視覺化呈現 (雷達圖)。 **批次處理**：夜間批次計算全公司落差數據。 | **負面感受**：過度強調「落差」可能打擊員工信心，UI 需強調「發展」。 **數據延遲**：若非即時計算，主管看到的可能是舊資料。 |
| **L2.6** | **AI 職能生成引擎** | **NLP 萃取**：從非結構化 JD 文字中自動提取 KSA 關鍵字。 **自動分類**：將提取出的技能自動歸類至職能庫。 | **語意歧義**：AI 可能將「溝通」誤判為「談判」，分類準確度需持續訓練。 **詞彙統一**：不同主管對同一技能的描述不同，易造成資料庫髒亂。 |

## **4\. L3 教育訓練模組 (Training & Development)**

| 編號 | 細項功能 | 開發重點 (Key Implementation) | 潛在風險與痛點 (Risks & Pain Points) |
| :---- | :---- | :---- | :---- |
| **L3.1** | **培訓計畫管理** | **預算控管**：實作預算扣減與凍結邏輯。 **版本歷程**：計畫變更需保留完整修訂紀錄。 | **流程僵化**：過於嚴格的簽核流程可能延誤培訓時機。 **預算超支**：若未整合請款系統，實際花費可能與計畫脫鉤。 |
| **L3.2** | **課程與報名管理** | **人才地圖渲染**：前端需優化 Heatmap 與 9-Box Grid 的大量 DOM 渲染效能。 **QR Code**：實作動態 QR Code 以防代簽到。 | **資料量大**：計算全公司人才地圖極耗資源，需預先計算 (Pre-aggregation)。 **代簽弊端**：即使有 QR Code，仍難完全杜絕截圖代簽，需搭配定位。 |
| **L3.3** | **線上測驗系統** | **防作弊**：偵測視窗切換 (Blur Event) 與禁止複製貼上。 **自動閱卷**：實作選擇題與簡答題 (AI 關鍵字比對) 評分。 | **瀏覽器相容性**：防作弊機制在不同瀏覽器行為可能不一致。 **網路中斷**：考試中斷線需有自動暫存與恢復機制。 |
| **L3.4** | **培訓成效追蹤** | **排程觸發**：精確執行 Course\_End\_Date \+ 90 的反饋啟動。 **雙向回饋**：實作「低轉化率 \-\> 標記課程」與「高轉化率 \-\> 升級職能」邏輯。 | **樣本回收率低**：三個月後員工多已遺忘，填寫意願低。 **成效難量化**：L3 行為改變難以單靠問卷證明，需主管深度參與。 |

## **5\. L4 專案管理模組 (Project Management)**

| 編號 | 細項功能 | 開發重點 (Key Implementation) | 潛在風險與痛點 (Risks & Pain Points) |
| :---- | :---- | :---- | :---- |
| **L4.1** | **專案與任務管理** | **遞迴結構**：資料庫需支援無限層級的 WBS (Project \-\> Phase \-\> Task)。 **隱形成本**：精確記錄 Duration \* Hourly\_Rate 並匯總。 | **UI 複雜度**：樹狀任務列表的拖曳與編輯互動開發難度高。 **工時填寫**：員工若不誠實記錄工時，隱形成本計算將完全失真。 |
| **L4.2** | **專案協作與進度** | **即時通訊**：使用 WebSocket 實作看板狀態同步與留言通知。 **甘特圖**：前端需實作複雜的時間軸渲染與相依性連線。 | **訊息轟炸**：過多的通知會造成干擾，需實作通知分級設定。 **效能問題**：大型專案任務過多時，甘特圖渲染可能卡頓。 |
| **L4.3** | **專案績效與毛利** | **混合預測**：整合 EVM 數學模型與 AI 情緒分析模型。 **預警系統**：當 Predicted\_Margin \< 0 時觸發即時 Alert。 | **誤報風險**：AI 可能對團隊的「自嘲」或「玩笑」誤判為負面風險。 **數據延遲**：財務數據若非即時入帳，毛利預測會有時間差。 |
| **L4.4** | **專案報表與分析** | **動態報表**：支援使用者自定義欄位與圖表類型。 **PDF 匯出**：後端需實作高解析度的報表生成服務。 | **報表跑版**：複雜圖表轉 PDF 常遇版面錯亂問題。 **計算負載**：產生大型報表時可能佔用大量 CPU 資源。 |
| **L4.5** | **專案自動化引擎** | **規則引擎**：讓使用者設定 If Condition Then Action (如：逾期 \-\> 通知)。 **排程服務**：高可靠性的 Job Queue (如 BullMQ) 處理自動化任務。 | **死循環**：錯誤的自動化規則可能導致系統無限發送通知。 **除錯困難**：自動化執行的 Log 需完整保留以便追查問題。 |

## **6\. L5 績效管理模組 (Performance Management)**

| 編號 | 細項功能 | 開發重點 (Key Implementation) | 潛在風險與痛點 (Risks & Pain Points) |
| :---- | :---- | :---- | :---- |
| **L5.1** | **過程管理與毛利** | **跨系統聚合**：從 ERP 獲取營收，從 L4 獲取成本，進行複雜運算。 **參數彈性**：需支援不同部門設定不同的間接成本分攤率。 | **計算錯誤**：財務計算容錯率為 0，任何小數點誤差都可能釀禍。 **鎖定機制**：計算後的數據必須 Immutable (不可變)，防止歷史被篡改。 |
| **L5.2** | **考核週期管理** | **批次處理**：考核啟動時需批次建立大量評分表單。 **狀態追蹤**：儀表板需即時顯示全公司考核完成率。 | **系統塞車**：考核截止日前夕大量用戶同時操作，需做負載測試。 **流程卡關**：若主管離職或請假，簽核流程需有代理人機制。 |
| **L5.3** | **360 度回饋系統** | **匿名邏輯**：確保前端與 API 回傳時移除評分者資訊。 **權限控管**：嚴格限制誰能看到原始評語。 | **匿名失效**：小團隊中透過語氣容易猜出評分者，導致不敢講真話。 **惡意攻擊**：需有機制過濾或標記帶有情緒性攻擊的評語。 |
| **L5.4** | **績效紀錄與日誌** | **CITA 整合**：將平時記錄的關鍵事件自動帶入考核表。 **全文檢索**：實作 ElasticSearch 以快速搜尋歷史日誌。 | **使用率低**：若操作不便，主管平時懶得記，考核時無據可查。 **隱私界線**：日誌內容是否對員工可見需有明確規範。 |
| **L5.5** | **績效分析與改善** | **校準機制 (Calibration)**：提供介面讓高層調整部門間的績效分佈曲線。 **PIP 追蹤**：針對低績效者自動建立改善計畫追蹤表。 | **強制分佈**：若系統強制常態分佈 (Bell Curve)，可能引發團隊不滿。 **標籤效應**：PIP 紀錄若未妥善保密，可能造成員工被貼標籤。 |

## **7\. L6 文化管理模組 (Culture Management)**

| 編號 | 細項功能 | 開發重點 (Key Implementation) | 潛在風險與痛點 (Risks & Pain Points) |
| :---- | :---- | :---- | :---- |
| **L6.1** | **企業文化手冊** | **CMS 系統**：實作簡易的內容管理系統，支援多媒體嵌入。 **EAP 匿名核銷**：開發 Hash ID \+ Token 的斷鏈驗證機制。 | **信任危機**：若員工懷疑 EAP 非真匿名，將導致服務無人使用。 **內容過時**：文化手冊若無定期維護提醒，易變成無效文件。 |
| **L6.2** | **獎項資料庫管理** | **爬蟲/API**：若能自動抓取外部獎項資訊更佳，或提供手動建檔介面。 **期限提醒**：實作多階段的截止日通知。 | **資料維護**：外部獎項規則變動頻繁，需人工定期確認。 **漏接風險**：通知信若被擋或忽略，可能錯過報名時機。 |
| **L6.3** | **文件儲存庫** | **分類標籤**：實作多維度的 Tag 系統。 **NAS 整合**：確保大檔案上傳的穩定性與斷點續傳。 | **儲存成本**：影音文件過多可能導致儲存空間與備份時間暴增。 **權限混亂**：資料夾權限若繼承邏輯複雜，易造成管理漏洞。 |
| **L6.4** | **AI 申請助理** | **RAG 技術**：利用 Retrieval-Augmented Generation 檢索內部數據生成申請書。 **多格式匯出**：支援 Word/PPT 格式生成。 | **內容空洞**：AI 生成的文案可能流於形式，缺乏靈魂與亮點。 **機密外洩**：需確保送給 LLM 的數據不包含敏感個資。 |
| **L6.5** | **智慧文件分析** | **OCR/NLP**：解析上傳的文件，檢查必要欄位是否缺漏。 **合規檢核**：比對文件內容是否符合獎項規範。 | **辨識率**：掃描檔或手寫文件的 OCR 辨識率可能不佳。 **規則維護**：合規規則需隨外部規範變動而更新，維護成本高。 |
| **L6.6** | **影響力評估引擎** | **統計模型**：實作迴歸分析 (Regression) 計算文化與績效的相關係數。 **視覺化報告**：生成易懂的影響力儀表板。 | **因果謬誤**：相關性不等於因果性，分析結果可能誤導決策。 **數據不足**：初期樣本數少時，統計結果不具顯著性。 |
