## 3. API Endpoints

| Method | Endpoint | Description |
| :--- | :--- | :--- |
| GET | `/api/jobs` | 取得職缺列表 (支援搜尋與篩選) |
| POST | `/api/jobs` | 新增職缺 (預設為 Draft) |
| GET | `/api/jobs/:id` | 取得單一職缺詳情 |
| PUT | `/api/jobs/:id` | 更新職缺 (若已同步則觸發 104 更新) |
| DELETE | `/api/jobs/:id` | 刪除職缺 (同步刪除 104 職缺) |
| PATCH | `/api/jobs/:id/status` | 更新狀態 (如發布/關閉，觸發 104 上下架) |
| POST | `/api/jobs/:id/sync-104` | 手動將職缺同步至 104 |
| POST | `/api/jobs/:id/sync-from-104` | 從 104 拉取最新資料回寫本地 |
| GET | `/api/jobs/:id/candidates` | 取得該職缺的所有應徵者 |

## 4. 業務邏輯與流程 (Workflow Logic)

1.  **職缺發布流程**:
    *   使用者在系統建立職缺，狀態為 `Draft`。
    *   填寫 104 相關欄位 (Salary, Location, Requirements)。
    *   將狀態改為 `Published` 時，系統自動呼叫 104 API (`postJob`)。
    *   若成功，回寫 `job104_no` 並將狀態設為 `104_synced`。

2.  **應徵者管理**:
    *   系統透過排程或 Webhook 接收 104 應徵者資料。
    *   資料寫入 `candidates` 表及相關子表 (`education`, `experience`...)。
    *   AI 模組對履歷進行解析與評分，更新 `score` 與 `ai_summary`。

3.  **資料同步策略**:
    *   **Local First**: 職缺編輯以本地資料為主，儲存後推送到 104。
    *   **Sync from 104**: 亦支援從 104 拉取資料覆蓋本地設定 (用於在 104 後台修改過的情況)。
