平台管理者、租戶與角色的關係

Bombus 採用 Database-per-Tenant 多租戶 SaaS 架構，以下定義各層級角色與上下關係。

平台管理員 (Platform Admin)
SaaS 營運者 — 管理所有租戶的生命週期
建立租戶 方案管理 平台審計
▼

super_admin 超級管理員
租戶內最高權限
hr_manager dept_manager employee
（完全獨立的資料庫）
平台層

獨立於租戶體系之外，使用 platform.db。專用登入端點，固定最高權限，無 Refresh Token。

租戶全域層

super_admin 為租戶內最高管理者，管理所有功能、帳號與角色。使用 tenant_*.db 獨立資料庫。

子公司 / 部門層

subsidiary_admin 管理特定子公司，dept_manager 管理部門，employee 為一般使用者。權限由 RBAC 矩陣控制。

功能權限矩陣

權限 = 「誰」可以對「哪些功能」做「什麼程度的操作」，看到「多大範圍的資料」

操作等級 — 能做什麼？

none（無權限）

完全看不到功能

view（僅查看）

只能看不能改

edit（可編輯）

完整讀寫權限

資料範圍 — 能看/改多大？

self（個人）

只能存取自己的資料

department（部門）

所屬部門全部資料

company（全公司）

整個租戶全部資料

重要規則：查看範圍 ≥ 編輯範圍（自動確保），調整編輯範圍時系統自動擴大查看範圍。
角色權限管理（續）

多角色權限合併：寬鬆策略

合併規則

等級排序：無權限 < 僅查看 < 可編輯
範圍排序：個人 < 部門 < 全公司

合併範例

角色A：L1.profile view/self + 角色B：L1.profile edit/department
↓
最終結果：edit / department

情境範例

陳經理 = 部門主管 + HR 經理

部門主管

員工檔案：edit / 個人（編輯）/ 部門（查看）

HR 經理

員工檔案：edit / 全公司 / 全公司

合併後

edit / 全公司 / 全公司 — 取最高等級 + 最廣範圍

一個使用者可有無限角色，系統自動取各角色最高權限。
角色權限管理（續）

常見操作情境

情境 A 新員工入職

使用者管理 → 確認帳號 → 指派「一般員工」角色（全域）

情境 B 升任部門主管

保留原角色 + 新增「部門主管」（作用範圍：行銷部）→ 權限自動合併

情境 C 建立自訂角色

新增角色「招募專員」→ 編輯功能權限（招募+面試+人才庫：edit/全公司）→ 指派

情境 D 確認使用者權限

角色管理 → 查看各角色權限 → 查看有效權限（合併後）

情境 E 暫時撤銷權限

方式一：停用帳號（推薦）
方式二：移除所有角色

情境 F 查看角色使用者

角色卡片 →「已指派 N 位」→ 彈出名單

認證與安全機制

登入方式

租戶登入

組織代碼 + Email + 密碼

POST /api/auth/login

平台管理登入

Email + 密碼

POST /api/auth/platform-login

JWT 雙 Token

Access Token 15 分鐘，localStorage，API 認證
Refresh Token 7 天，localStorage，自動刷新

自動刷新流程：

API 請求 → 401 → 攔截 → 有 Refresh? → 刷新 + 重送

密碼安全

加密：bcryptjs（salt: 10）
最小長度：8 字元
變更後：撤銷所有 Refresh Token
首次登入：must_change_password 強制修改
