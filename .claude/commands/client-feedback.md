# 客戶回饋文件 → 系統比對分析報表

將客戶提供的修改建議文件（**.docx 或 .pdf**）自動拆解為問題項，與系統現況進行比對分析，產出標準格式的 xlsx 報表。

## 輸入參數

`$ARGUMENTS` 應包含：

1. **文件路徑**（必填）：客戶回饋文件路徑（支援 `.docx` 或 `.pdf`）
2. **輸出路徑**（選填）：指定輸出 xlsx 路徑。預設為 `bombus-system/docs/客戶回饋比對分析_<原檔名>_YYYYMMDD.xlsx`
3. **比對範圍**（選填）：限定比對的模組範圍（如 `L1-招募` `L2-職能`），預設全部

---

## 執行流程

### Phase 1：解析客戶文件

#### 1.1 依副檔名選擇解析路徑

| 副檔名 | 解析方式 | 工具 |
|--------|---------|------|
| `.docx` | 使用 python-docx 提取段落、表格、粗體標記、圖片占位 | `from docx import Document` |
| `.pdf` | 直接使用 Claude Code 內建 `Read` 工具（原生支援 PDF，能同時看到文字 + 圖片） | `Read(file_path=..., pages="1-N")` |
| `.PDF`（大寫） | 同上 | 同上 |

> **PDF 解析重點**：Claude `Read` 工具一次最多讀 20 頁。文件超過 10 頁時**必須**指定 `pages` 範圍分批讀取（如先 `1-10`，再 `11-20`）。`Read` 對 PDF 的回傳是文字 + 視覺渲染，圖片資訊不會遺失，但**文字結構（粗體 / 標題層級）需從排版上下文推斷**，不像 docx 有明確的 `bold`/`heading` 屬性。

#### 1.2 辨識文件結構

不論 docx 或 pdf，都需識別：

- **模組標題**：以「L0」「L1」「L2」「第一部分」「第二部分」等開頭，或顯著大字粗體
- **章節編號**：`1.` `2.` `3.` 開頭的功能分組，或 `(一)` `(二)`
- **問題描述段落**：項目符號（• – ·）後的內容段落
- **流程描述**：含 `→` `->` `=>` 的句子
- **強調標記**：
  - docx：使用 `run.bold` 屬性判定
  - pdf：從排版上下文（重複出現的關鍵詞、句首標籤）推斷
- **介面參考圖片**：標記為「附圖N（參見原始文件第 N 頁）」

#### 1.3 問題拆解

將連續段落歸類為獨立問題項，每項包含：
- 所屬模組（L0/L1/L2…）
- 功能分類（自動歸納的 3~8 字短標題）
- 原始需求內容（保留客戶用語）
- 相關流程描述（如有）
- 來源頁碼（PDF）或段落索引（docx）— 方便後續回溯

#### 1.4 向使用者確認

輸出拆解結果摘要表（`| 編號 | 模組 | 功能分類 | 一句話描述 |`），確認問題項數量與分類正確後再進入 Phase 2。

---

### Phase 2：平行系統比對

針對每個問題項，啟動 Agent（`subagent_type=Explore`）平行探索：

**按模組分組啟動探索代理（最多 3~4 個並行）**：

每個代理的探索重點：
- 相關元件/頁面的現有實作狀態
- 相關 API 端點的實作狀態
- 相關 DB schema 是否支援
- 現有 UI 的 Tab/按鈕/流程與需求的差異

探索結果需包含：
- 具體檔案路徑與行號（使用 `file:line` 格式）
- 實作狀態判定（✅ 已實作 / ⚠️ 部分實作 / ❌ 未實作）
- 差異描述

---

### Phase 3：產出需求理解與分析

對每個問題項填寫完整分析：

| 欄位 | 說明 | 填寫指引 |
|------|------|----------|
| 編號 | D-01, D-02... | 依序編號 |
| 模組 | L0-xxx / L1-xxx / L2-xxx | 從文件標題識別 |
| 功能分類 | 3~8 字短標題 | 歸納問題的核心功能點 |
| 修訂需求內容 | 客戶原始描述 | 保留原意，精簡贅詞，合併相關段落 |
| 需求理解 | 技術語言轉譯 | 將客戶描述轉為可執行的技術需求 |
| 執行狀態 | ✅/⚠️/❌ | 基於 Phase 2 探索結果判定 |
| 執行現況 | 詳細系統現況 | 包含檔案路徑、已有/缺失功能、具體程式碼引用 |
| 影響範圍 | 相關檔案路徑 | 需修改的前端元件/後端路由/DB schema |
| 風險點評估 | 潛在風險 | 改動影響範圍、相依性、資料一致性風險 |
| 需要討論的問題 | 待釐清事項 | 需求模糊處、技術方案選擇、需客戶確認的問題 |
| 修改難易度 | 低 / 中 / 高 | **必填**。低=純設定/小幅 UI；中=新增 schema 欄位+API；高=跨層大改/新模組/破壞既有資料模型。已實作（✅）填「—」 |
| 修改狀態 | 尚未修正 / 進行中 / 已修正 | 初版分析全部填「尚未修正」（已實作項填「—」）。後續 PR/實作完成時改為「已修正」 |
| 預計修改 | 討論收斂後的設計方案（內部技術紀錄） | 初版可空白。經 `/spectra:discuss` 收斂後填入：schema 草案、API 端點、UI 流程、雙清單同步點、難易度上修建議等 |
| 修改說明 | 實作完成後的變更紀錄（內部技術紀錄） | 初版可空白。實作完成時填寫變更摘要 + commit hash 或 OpenSpec change 名稱 |
| 預計提供測試時間 | 客戶可進入測試的日期（客戶面） | 初版**可留空**。實作排程確認後填入（如「2026-05-15 起可進入測試」或「需先討論收斂後 +2-3 週」）。✅ 已實作項已在系統中可填「現可測試（已部署主線）」，現行已具備 / 不納入本次調整項填 `—` |
| 預計處理說明 | 客戶面非技術描述（客戶面） | **必填**（除「現行已具備」可寫「現行已有 + 一句話功能描述」）。避技術詞、保留難度感。詳見下方「預計處理說明撰寫規範」 |

> **欄位順序固定為 16 欄**（2026-05-08 起由 14 欄升級），詳見 memory `feedback_xlsx_16col_standard.md`。13/14 欄為內部技術紀錄，15/16 欄為客戶面溝通欄。

### 預計處理說明撰寫規範（必讀）

此欄位是**寄給客戶的文件**內容，撰寫原則：

**禁用技術詞**（客戶看不懂或誤判）：

| ❌ 不可用 | ✓ 改寫為業務語 |
|---|---|
| spec / proposal / OpenSpec change | 設計方案 / 規劃內容 |
| schema / migration / ALTER TABLE | 資料結構 / 資料表變更 |
| index / 索引 SQL | 資料庫索引 |
| signal / Angular / middleware / RxJS | （直接省略，描述前端行為即可） |
| API / endpoint / route | 後端服務 / 系統介面 |
| commit / PR / branch | （省略，描述系統變更即可） |
| Refactor / N+1 / debounce | 重整 / 整體調整（省略技術細節） |

**保留難度感**（避免客戶誤判為「不過是加個按鈕」）：

- 條列影響範圍（用業務術語：列表頁、設定頁、員工檔案頁，不寫 component name）
- 列出處理重點 2-4 點，可提「資料查詢邏輯」「資料庫索引」「設定介面」這類業務可理解的概念詞
- 提及「需確保既有功能不受影響」「同步維護新／舊資料」之類，傳達需要審慎處理的訊號
- 結尾說明「完成後您可以...」（具體益處）或「預先需討論...」（待決議題）

**狀態對應寫法**：

| 執行狀態 | 預計處理說明寫法 |
|---|---|
| ✅ 已實作 | 「現行已有」+ 一句話業務功能描述 |
| ⚠️ 部分實作 | 「擴充 X 功能：⋯」+ 影響範圍 + 處理重點 + 預計效果 |
| ❌ 未實作 | 「新增 X 功能：⋯」+ 影響範圍 + 處理重點 + 預計效果（或「預先需討論」議題） |
| 不納入本次 | 「原規劃不包含 X⋯，需後續評估實際執行方式與需求」（軟著陸句必加） |

**範例（D-13 員工大表分頁）**：

```text
員工列表頁面整體升級：新增「分頁瀏覽 + 搜尋 + 欄位排序」功能。

• 影響範圍：兩處員工列表（系統設定 → 員工與帳號管理、員工檔案頁）。

• 處理重點：
  ① 調整後端員工查詢邏輯，新增分頁、搜尋、排序的支援；
  ② 重整兩個列表頁面 UI，確保既有的部門/狀態/權限範圍篩選不受影響；
  ③ 新增資料庫索引以維持大量員工下的查詢速度；

• 完成後您可以：以姓名／員編／Email 直接搜尋員工、點欄頭依到職日/部門/姓名排序、依需要切換每頁筆數（20/50/100/200），員工人數成長後依然順暢。
```

### Phase 4：產出 xlsx 報表

使用 openpyxl 產出三個工作表的 xlsx：

#### Sheet 1：修訂需求總表

- **16 欄標準格式**（如上表，編號→修訂需求內容→⋯→修改說明→預計提供測試時間→預計處理說明）
- 標題列：深藍底白字、粗體
- 執行狀態欄位顏色：✅ 綠底 / ⚠️ 黃底 / ❌ 紅底
- 修改難易度欄位顏色：低=淡綠 (E2EFDA) / 中=淡黃 (FFF2CC) / 高=淡紅 (FCE4D6)
- 凍結首列、啟用自動篩選
- 欄寬自適應內容（建議 16 欄寬：6, 18, 20, 50, 50, 12, 55, 40, 40, 40, 12, 14, 40, 40, 28, 50）

#### Sheet 2：圖例說明
- 狀態顏色圖例（✅/⚠️/❌）
- 編號區間對應模組
- **新增**：來源檔案類型（docx / pdf）與頁數標註

#### Sheet 3：統計總覽
- 各模組的 ✅/⚠️/❌ 數量統計
- 合計列

#### Sheet 4（選填）：非功能需求附錄

- 編號使用 `ND-XX` 系列（與主表 `D-XX` 區分）
- 欄位：編號 / 類別 / 需求內容 / 目前狀態 / 建議行動 / 對應功能編號
- 用於存放純效能、架構、開發提醒類項目（非可勾選的功能需求）

### Phase 5：xlsx 結構化驗證（必跑）

**產出 xlsx 後不可直接回報完成**。必須跑一支驗證腳本核對輸出，**全部通過才能進入 Phase 6 回報**。

#### 5.1 驗證腳本骨架

於 `/tmp/verify_<原檔名>_report.py` 寫驗證腳本（每次任務新建一支，避免汙染專案目錄）：

```python
from openpyxl import load_workbook
from collections import Counter, defaultdict

PATH = "<xlsx 輸出路徑>"

# 期望值（從 Phase 3 整合的資料填入）
EXPECTED_SHEETS = ["修訂需求總表", "圖例說明", "統計總覽", ...]
EXPECTED_HEADERS = ["編號","模組","功能分類","修訂需求內容","需求理解",
                    "執行狀態","執行現況","影響範圍","風險點評估","需要討論的問題",
                    "修改難易度","修改狀態","預計修改","修改說明",
                    "預計提供測試時間","預計處理說明"]
EXPECTED_STATUS = {"D-01": ("L0-RBAC", "✅"), ...}  # 主表逐列預期值
EXPECTED_DIFFICULTY = {"D-01": "—", "D-02": "高", ...}  # 修改難易度逐列預期值

REPORTED_DONE = X        # 即將在 Phase 6 回報的 ✅ 數
REPORTED_PARTIAL = Y     # 即將回報的 ⚠️ 數
REPORTED_NOT = Z         # 即將回報的 ❌ 數
```

#### 5.2 必跑驗證項目（17 項）

| # | 驗證項 | 通過條件 |
|---|--------|---------|
| 1 | 工作表結構 | 實際 sheetnames == 期望 |
| 2 | 標題列完整 | 主表 **16 欄**標題與規範一致（含修改難易度/修改狀態/預計修改/修改說明/預計提供測試時間/預計處理說明） |
| 3 | 資料列數 | 與 Phase 3 整合的問題數一致 |
| 4 | 編號連續性 | `D-01 ~ D-NN` 無斷號、無重複 |
| 5 | 模組×狀態核對 | 逐列比對 `EXPECTED_STATUS`，全部相符 |
| 6 | 必填欄位非空 | 修訂需求/需求理解/執行現況/影響範圍/風險/討論 6 欄皆有實質內容（>5 字） |
| 7 | 狀態著色正確 | ✅→C6EFCE / ⚠️→FFEB9C / ❌→FFC7CE，全列符合 |
| 8 | 凍結窗格 | `freeze_panes == "A2"` |
| 9 | 自動篩選 | `auto_filter.ref` 涵蓋整張資料區 |
| 10 | 統計總覽交叉核對 | 各模組 ✅/⚠️/❌ 數量 == 主表逐列計數（zero-tolerance） |
| 11 | 合計列正確 | 合計 == 各模組加總 == 主表總列數 |
| 12 | 回報數字一致 | `REPORTED_DONE/PARTIAL/NOT` == 主表計數（防止文字回報與檔案不一致） |
| 13 | 修改難易度必填 | 每列「修改難易度」欄位值 ∈ {低, 中, 高, —}；未實作（❌）/部分實作（⚠️）必為 低/中/高，不可空白 |
| 14 | 統計含難易度欄 | 「統計總覽」需含「難易度:低/中/高」三欄，加總 == 主表逐列計數 |
| 15 | 預計處理說明非空 | 每列「預計處理說明」（col 16）須有實質內容（>10 字）。✅ 列可寫「現行已有 + 功能描述」；⚠️/❌ 列必含影響範圍 + 處理重點 |
| 16 | 預計處理說明禁用詞 | 每列 col 16 不得含技術術語：spec / migration / schema / index 索引欄位 / signal / Angular / middleware / SQL / commit / PR / API / endpoint。出現任一詞 → 失敗 |
| 17 | 預計提供測試時間格式 | 「預計提供測試時間」（col 15）若有值，須為日期格式（YYYY-MM-DD / M/D）或描述句（如「需先討論⋯」）；空白或 `—` 也允許 |

#### 5.3 失敗處理

- **任一項不通過 → 不可進入 Phase 6**。先修 xlsx 產出腳本（`/tmp/build_*.py`），重跑直到驗證腳本以 exit code 0 結束
- 不要用「警告當通過」— 顏色錯、編號跳號、統計不一致都是硬性失敗
- 內容過短（<5 字）視為失敗 — 表示 Phase 3 整合不完整

#### 5.4 通過後輸出

驗證腳本應輸出類似：

```text
✓✓✓ 全部通過：xlsx 內容正確無誤

模組          ✅   ⚠️   ❌   合計
L0-RBAC       1    3    1    5    ✓
...
合計          1   12    4   17    ✓
主表計數 = 統計總覽 = 合計列 = 回報數字（三方一致）
```

## xlsx 格式規範

### 樣式定義

```python
# 標題列
header_font = Font(bold=True, size=11, color="FFFFFF")
header_fill = PatternFill(start_color="2F5496", end_color="2F5496", fill_type="solid")

# 狀態顏色
fill_done = PatternFill(start_color="C6EFCE", fill_type="solid")      # ✅
fill_partial = PatternFill(start_color="FFEB9C", fill_type="solid")    # ⚠️
fill_not = PatternFill(start_color="FFC7CE", fill_type="solid")        # ❌

# 欄寬建議
col_widths = [6, 18, 20, 50, 50, 12, 55, 40, 40, 40, 12, 14, 40, 40, 28, 50]
```

### 內容規範

- **修訂需求內容**：保留客戶原意，去除口語贅詞，合併同主題段落
- **需求理解**：用技術語言重新詮釋，讓開發者能直接理解要做什麼
- **執行現況**：以「狀態判定 + 具體說明」格式，引用檔案路徑與行號
- **影響範圍**：列出所有需要修改的檔案路徑（一行一個）
- **風險點評估**：聚焦改動的副作用、相依性、資料遷移風險
- **需要討論的問題**：以問句形式，列出需要客戶或團隊確認的事項

## 注意事項

### 通用

- 文件中的圖片無法直接提取為文字，以「附圖（參見原始文件第 N 頁）」標註
- 若文件內容涉及多個模組，平行探索時按模組分組（每組一個 Agent）
- 產出的 xlsx 可直接作為 `/spectra:propose` 的輸入依據
- 每個問題項的需求理解應足夠具體，讓開發者不需回頭看原始文件

### docx 專屬

- 若客戶文件使用「客戶回饋範本」格式，可直接對應表格欄位，加速解析
- `python-docx` 的 `Document.paragraphs` 已包含順序資訊，搭配 `paragraph.style.name` 判定標題層級

### pdf 專屬

- **大檔案分批讀取**：超過 10 頁時必須使用 `pages="1-10"`、`pages="11-20"` 分批，否則 `Read` 會失敗
- **PRD / 需求確認單**格式：通常是「模組資訊 → 功能需求（分章節）→ 例外處理 → 開發提醒」結構，可預期會看到圖片佔大量版面
- **無法依賴文字粗體屬性**：`Read` PDF 看到的是視覺渲染，需從重複關鍵詞、冒號前的標籤、上下文位置判斷強調
- **頁碼必填**：每個拆解出的問題項要記錄來源頁碼，方便客戶 review 時對照原文
- **避免使用第三方 PDF 庫**：本機 Python 預設無 `pymupdf` / `pdfplumber` / `PyPDF2`，直接走 Claude `Read` 工具，無需額外安裝
