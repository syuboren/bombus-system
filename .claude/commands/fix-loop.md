## 自主測試驅動修復迴圈（Fix Loop）

你將進入一個**自主修復迴圈**，規則如下：

### 輸入

使用者會描述一個 bug 或提供失敗的測試。如果沒有明確描述，先問清楚：
- 什麼行為是錯的？
- 預期行為是什麼？
- 哪個模組/頁面受影響？

### 步驟 1：重現問題

在修任何東西之前，**先重現失敗**：

- 如果有整合測試相關，執行它：
  ```bash
  cd bombus-system/server && node src/tests/test-e2e-flow.js
  ```
- 如果是前端問題，執行型別檢查和建置：
  ```bash
  cd bombus-system && npx tsc --noEmit && npx ng build --configuration=development
  ```
- 記錄失敗的具體錯誤訊息

如果無法重現問題，先調查根因再繼續。**不要盲目修改。**

### 步驟 2：根因分析

在動手修之前，先分析根因：

1. 讀取相關程式碼檔案
2. 檢查 DB 欄位是否存在、ID 命名是否一致
3. 確認前端欄位 → API payload → DB 欄位的對應鏈路
4. 檢查 `tenant-schema.js` 和 `tenant-db-manager.js` 遷移清單是否同步
5. 簡要說明你認為的根因

### 步驟 3：修復迴圈（最多 5 輪）

```
FOR round = 1 TO 5:
    1. 套用最小修復（只改必要的部分）
    2. 立即重新執行步驟 1 的測試
    3. IF 全部通過 → 跳到步驟 4
    4. IF 你的修復引入新的失敗：
       - 記錄新的失敗
       - 評估：是修復不完整，還是方向錯誤？
       - 如果方向錯誤 → 復原此輪修改，嘗試不同方法
       - 如果修復不完整 → 繼續修
    5. 下一輪
```

**關鍵規則：**
- 每輪修復後**必須**重跑測試，不得跳過
- 如果同一個方法連續失敗 2 輪，切換策略
- 如果 5 輪後仍未解決，停止並報告分析結果，讓使用者決定

### 步驟 4：完整驗證

所有測試通過後，執行完整驗證：

```bash
# 型別檢查
cd bombus-system && npx tsc --noEmit

# 建置
cd bombus-system && npx ng build --configuration=development

# 相關整合測試（如適用）
cd bombus-system/server && node src/tests/test-e2e-flow.js
```

### 步驟 5：報告

```
## 修復報告

**問題：** [描述]
**根因：** [分析]
**修復：** [做了什麼]
**修改檔案：** [列出]
**驗證：**
  ✅ 型別檢查通過
  ✅ 建置通過
  ✅ 整合測試通過（N/N）
**修復輪數：** M/5
```

### 禁止事項

- 不得在未重跑測試的情況下宣告修復完成
- 不得忽略修復引入的新失敗
- 不得一次修改過多檔案 — 保持修復最小化
- 不得跳過根因分析直接猜測修改
